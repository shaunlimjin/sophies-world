# Sophie's World

A weekly HTML email newsletter for Sophie (4th-grade reading level), generated each week and delivered via Gmail.

Each issue mixes kid-friendly world news, fun facts, a Singapore ↔ USA cultural link, an interest section (currently Gymnastics Corner), a money/entrepreneurship lesson, and a math challenge tied to the week's news.

Content is generated by hosted models and rendered locally into HTML from a structured JSON artifact. The app now uses a split config architecture (`children/`, `sections/`, `pipelines/`) plus a lightweight staging/approach system for safe experimentation before changes reach prod.

---

## Quick start

```bash
pip3 install -r requirements.txt            # install deps (PyYAML)
cp .env.example .env                        # then fill in API keys
python3 scripts/generate.py --test          # generate a test issue
open newsletters/test/sophies-world-$(date +%F).html
```

To send the real weekly issue:

```bash
python3 scripts/generate.py && python3 scripts/send.py
```

`scripts/run.sh` wraps this for cron and appends to `logs/run.log`.

## What's in each issue

- **Weird But True** — 2–3 wild fun facts
- **World Watch** — 2 current events explained simply, including serious topics
- **Singapore Spotlight** — culture, history, food, nature
- **USA Corner** — California/Fremont angle or US sports/science/culture
- **Gymnastics Corner** — interest section (swappable via config)
- **Money Moves** — saving/entrepreneurship concept + real kid-entrepreneur story
- **Sophie's Challenge** — math/reasoning puzzle tied to that week's World Watch

The active set and ordering live in `config/children/sophie.yaml` (`newsletter.active_sections`).

## How it works

At the engine level, the app still has two execution paths:

```text
resolved child + section + pipeline config
   │
   ├── Mode A ───────────── content_stage.py (hosted with integrated search)
   │
   └── Mode B ── research_stage.py → ranking_stage.py → content_stage.py
                 (Brave retrieval)   (prefilter + rank)  (packet synthesis)
   │
   ▼
issue_schema.py  (validate + persist structured issue artifact)
   │
   ▼
render_stage.py  (deterministic local HTML render into template.html)
   │
   ▼
newsletters/...html  →  send.py (Gmail SMTP)
```

In practice, though, we usually do **not** think in terms of hand-picking Mode A/B on every run.
The more durable operating model is:
- `config/pipelines/default.yaml` defines the default provider/model routing
- staging and approach overlays define experimental variants
- direct provider CLI flags still exist, but they are mainly for ad hoc debugging/testing rather than the preferred way to encode lasting experiments

### Generation patterns

| Pattern | `content_provider` | `ranker_provider` | What it does |
|---|---|---|---|
| A | `hosted_integrated_search` | — | Hosted model generates the issue directly with integrated search |
| B1 | `hosted_packet_synthesis` | `heuristic_ranker` | Brave retrieval → heuristic ranking → hosted packet synthesis |
| B2 | `hosted_packet_synthesis` | `hosted_model_ranker` | Brave retrieval → model reranking → hosted packet synthesis |

These are best treated as **conceptual execution patterns**, not the main operator interface.
The actual selected behavior comes from:
- `config/pipelines/default.yaml`
- optional staging overrides
- optional approach overrides

**Current recommendation: B1-style execution** as the default baseline. B2 is still experimental and has not yet clearly beaten B1 on coherence + novelty.

### Preferred workflow: encode variants as approaches

If we want a durable experiment, create or update an approach and specify its provider/model config there instead of relying on a long CLI command.

Examples:

```bash
# Run the prod baseline defined by config/pipelines/default.yaml
python3 scripts/generate.py --test

# Run staging baseline (same structure, isolated outputs)
python3 scripts/generate.py --env staging --test

# Run a durable experimental approach
python3 scripts/generate.py --env staging --approach approach-b1 --test
```

Examples of what an approach can own:
- different `content_provider`
- different `ranker_provider`
- different synthesis/ranking model choices
- prompt or script changes under `staging/approaches/<name>/scripts/`

### CLI overrides: still available, but debug-only in spirit

```bash
# One-off direct test of integrated-search generation
python3 scripts/generate.py --test --content-provider hosted_integrated_search

# One-off direct test of heuristic packet synthesis
python3 scripts/generate.py --test --content-provider hosted_packet_synthesis --ranker heuristic_ranker

# One-off direct test of model-ranked packet synthesis
python3 scripts/generate.py --test --content-provider hosted_packet_synthesis --ranker hosted_model_ranker

# Refresh the Brave research packet for a one-off test
python3 scripts/generate.py --test --refresh-research

# Compare two ad hoc runs side-by-side without overwriting
python3 scripts/generate.py --test --run-tag mode-a --content-provider hosted_integrated_search
python3 scripts/generate.py --test --run-tag mode-b1 --content-provider hosted_packet_synthesis --ranker heuristic_ranker
```

These are useful for spot checks, debugging, and evaluation runs. They are **not** the preferred way to encode durable operating modes — use pipeline config + staging/approach overlays for that.

`--run-tag <name>` appends the tag to the HTML, issue artifact, and (for Mode B) research packet filenames.

### Example: what this means in practice

- Want a lasting experiment with a different provider/model mix? → make or update an **approach**.
- Want a quick spot check right now? → use a **CLI override**.
- Want the normal weekly run? → rely on the **pipeline default**.

### Research packet caching

Mode B writes research packets to `artifacts/research/` keyed by a `config_hash`.
In current code, that hash is a fingerprint of the **retrieval-shaping inputs**, not the full resolved config tree. It currently includes:
- active section IDs
- per-section `research.queries`
- per-section `research.count`
- per-section `research.freshness`

Behavior:
- hash matches → reuse the cached packet
- hash mismatches → rerun research/ranking automatically
- `--refresh-research` → always rerun

This lets you iterate on synthesis prompts or ranking without burning Brave API quota, while still invalidating cached packets when retrieval inputs change.

## Environments and approaches

The environment overlay system still works the same way after the config refactor:
- prod baseline lives in `config/`
- staging overrides live in `staging/config/`
- named approach overrides live in `staging/approaches/<name>/config/`

Prod, staging, and `approach-b1` now all have the split config structure in place (`children/`, `sections/`, `pipelines/`). Legacy monolithic files still remain temporarily for safe transition/testing and are tracked for later deletion.

The app has two runtimes: **prod** (what Sophie actually receives) and **staging** (isolated, for testing changes). Inside staging, named **approaches** let you test fully self-contained variants.

### Environment selection

```bash
python3 scripts/generate.py --test                                           # prod (default)
python3 scripts/generate.py --env staging --test                             # staging
python3 scripts/generate.py --env staging --approach approach-b1 --test      # staging + approach
```

### Config overlay

Each environment resolves config files through a layered fallback:

| Env | Resolution order |
|---|---|
| `prod` | `config/` |
| `staging` | `staging/config/` → `config/` |
| `staging --approach <name>` | `staging/approaches/<name>/config/` → `staging/config/` → `config/` |

Only files you want to override need to exist in the staging layer; everything else falls back to prod.

### Output routing

| Invocation | HTML | Artifacts |
|---|---|---|
| `prod` | `newsletters/` | `artifacts/` |
| `prod --test` | `newsletters/test/` | `artifacts/` |
| `staging` | `newsletters/staging/` | `artifacts/staging/` |
| `staging --approach <name>` | `artifacts/approaches/<name>/newsletters/` | `artifacts/approaches/<name>/` |

### Promoting changes

```bash
python3 scripts/promote.py --from approach-b2-v2 --to staging     # approach → staging
python3 scripts/promote.py --from staging --to prod               # staging → prod
python3 scripts/promote.py --from staging --to prod --dry-run     # preview only
python3 scripts/promote.py --from staging --to prod --yes         # skip confirmation
```

Promotions show a diff, require confirmation (unless `--yes`), and auto-commit the result. Promoting an approach directly to prod is blocked — it must go through staging first.

### Creating a new approach

Duplicate `staging/approaches/approach-b1/` as a starting point, edit its `config/` and/or `scripts/`, then run it with `--env staging --approach <your-approach>`.

## Config

| File / dir | What it controls |
|---|---|
| `config/children/sophie.yaml` | child identity, interests, active sections, child-level editorial defaults |
| `config/sections/*.yaml` | one file per section: display metadata, editorial goal/rules, research config, section-local ranking intent |
| `config/pipelines/default.yaml` | provider routing, model selection, global domains, global ranking defaults, novelty settings |
| `config/themes/default.yaml` | template path + section ordering |
| `config/research.yaml` / `config/sections.yaml` | legacy monolithic compatibility files still present temporarily; tracked for deletion after transition verification |

To swap the interest section (e.g. Gymnastics Corner → K-pop Corner): edit `newsletter.active_sections` in `config/children/sophie.yaml` so the ID matches a file in `config/sections/`, then regenerate.

## Setup

**Requirements:** Python 3, Brave Search API key (for Mode B), Gmail account with app password, and whatever hosted model credentials are needed by the selected provider configuration.

**Environment** — copy `.env.example` to `.env` and fill in:

```env
BRAVE_API_KEY=...
GMAIL_USER=your.gmail@gmail.com
GMAIL_APP_PASSWORD=xxxx xxxx xxxx xxxx
RECIPIENT_EMAIL=recipient@example.com
```

**Automation** — a cron job on a Mac Mini runs `scripts/run.sh` every Saturday at 6am Pacific.

## Known limitations

- `generate.py` still loads `config/children/sophie.yaml` directly — multi-child selection (e.g. `--child`) is not yet exposed
- issue numbering is derived from file count (fine for now, not fully robust)
- split config is now present across prod, staging, and `approach-b1`, but legacy monolithic compatibility files still remain pending cleanup
- research cache keys on config shape, not on broader external retrieval state
- source-link quality is not yet production-grade; generated newsletters can still emit generic, duplicated, or weakly grounded links unless we add explicit validation/grounding checks
- `artifacts/` is local-only and fully gitignored
