## Correction
This project is **not** a UI app. The earlier test draft leaned too far into app-style QA. This revised plan tests the thing that actually exists: the newsletter generation pipeline, run artifacts, MiniMax execution paths, and config resolution.

## Goal
Sanity-check that the workflow built over the past few days actually works before moving on to experiments or model comparisons.

## Scope
We are testing three things:

1. **Run lifecycle**
   - create a run
   - inspect current run output/status
   - view/list prior runs via artifacts/logs/output files

2. **MiniMax execution**
   - **Mode A**: full newsletter generation using hosted integrated search
   - Stage-level execution / pipeline behavior for:
     - research
     - rank
     - synthesis

3. **Re-architected config behavior**
   - split config architecture resolves correctly (`children/`, `sections/`, `pipelines/`)
   - staging / approach overlays route correctly
   - invalid or missing config fails clearly

## Out of scope
- Broad model quality comparisons
- Prompt experimentation beyond basic sanity checks
- Gmail send-path validation unless needed later
- Long-form evaluation of newsletter quality beyond “usable / structurally correct / obviously broken”

## Exit criteria
We can move on to experiments once:
- a normal test run can be generated successfully
- prior runs/artifacts can be inspected reliably
- MiniMax Mode A works end-to-end
- stage pipeline behavior is validated enough to trust research → rank → synthesis wiring
- config refactor behaves as intended in tested scenarios
- failures are either fixed or documented with clean repro steps

## Key commands / surfaces
Primary generation entrypoint:
```bash
python3 scripts/generate.py --test
```

Useful variants:
```bash
python3 scripts/generate.py --test --content-provider hosted_integrated_search
python3 scripts/generate.py --test --content-provider hosted_packet_synthesis --ranker heuristic_ranker
python3 scripts/generate.py --test --content-provider hosted_packet_synthesis --ranker hosted_model_ranker
python3 scripts/generate.py --env staging --test
python3 scripts/generate.py --env staging --approach approach-b1 --test
python3 scripts/generate.py --test --refresh-research
```

Primary evidence locations:
- `newsletters/`
- `newsletters/test/`
- `artifacts/`
- `artifacts/research/`
- `artifacts/staging/`
- `artifacts/approaches/<name>/`
- `logs/`

## Execution notes

- capture the exact command used
- capture stdout/stderr or the most relevant log excerpt

- record the exact output locations in the report so artifacts/issues are easy to open later
- mark **Pass**, **Fail**, or **Blocked**
- for Fail/Blocked include:
  - repro steps

- whether the output came from `prod --test`, `staging`, or `staging --approach <name>`

## Pass 3 — Regression / sanity

### T16. Generated results are reviewable from artifacts
**Purpose:** ensure output + artifacts + logs tell one coherent story.

**Steps**
1. run one successful generation
2. inspect newsletter output
3. inspect corresponding artifact(s)
4. inspect relevant log lines

**Expected**
- output, artifact, and logs line up cleanly
- easy to tell what happened in the run

### T17. Basic repeatability
**Purpose:** rule out “works once” bugs.

**Steps**
1. repeat one default test run
2. repeat one Mode A run
3. repeat one staging or approach run

**Expected**
- repeated runs also work
- no obvious temp-state poisoning or caching weirdness beyond expected research caching

### T18. Research cache behavior is sane
**Purpose:** verify research packet caching after config refactor behaves as documented.

**Steps**
1. run a packet-synthesis path twice without `--refresh-research`
2. verify cached packet reuse
3. rerun with `--refresh-research`

**Expected**
- cache is reused when inputs match
- research reruns when explicitly refreshed
- behavior matches README description
